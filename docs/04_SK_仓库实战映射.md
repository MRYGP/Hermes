# SK 仓库实战映射

这份文档用于回答：

> Hermes / Agent OS 的概念，在 SK 仓库里到底拿来干什么？

## 三条主链路

### 1. 状态维护链路

目标：

- 发现文档状态漂移
- 保持看板、任务、产物之间一致
- 避免旧结论被当成新事实

适合沉淀为：

- skill：状态漂移审计
- quick command：读取关键状态文件
- automation：定期只读审计

第一版动作：

> 写一个“状态漂移审计”模板，只输出问题清单，不自动改文件。

### 2. 产物发布链路

目标：

- 把阶段成果整理成发布包
- 检查文档是否完整
- 生成摘要、变更说明、验收清单

适合沉淀为：

- skill：发布包生成
- quick command：生成固定格式 release note
- hook：发布前检查关键文件是否存在

第一版动作：

> 先用模板生成发布包清单，再由 Codex 执行文件整理。

### 3. 学习复盘链路

目标：

- 记录学到了什么
- 判断哪些机制值得迁移到 Codex / Claude Code
- 形成下一步学习计划

适合沉淀为：

- ChatGPT Project 指令
- 复盘模板
- 知识库稳定文件

第一版动作：

> 每天只记录 3 件事：今天学的概念、在 SK 里的用途、下一步动作。

## 什么适合做成 skill

适合做成 skill 的动作应该满足：

- 会重复执行
- 有明确输入输出
- 有固定步骤
- 需要少量判断但不需要完全开放推理

候选：

- 状态漂移审计
- 收尾联动检查
- 发布包生成
- 项目复盘
- 文档过期检查

## 什么适合做成 quick command

适合做成 quick command 的动作应该满足：

- 高频
- 短小
- 机械
- 输出格式固定

候选：

- 打开项目总览
- 生成今日复盘模板
- 列出待审计文件
- 输出固定 commit message 草稿

## 什么适合做成 cron / automation

适合周期运行的动作应该先满足：

- 只读优先
- 低风险
- 可审计
- 输出清晰

候选：

- 每日状态漂移检查
- 每周知识库过期检查
- 每周工具分工复盘
- 发布前文档完整性检查

## 第一阶段不要自动化的事

- 自动重写长文
- 自动改项目状态
- 自动发布
- 自动删除或移动文件
- 自动修改关键配置

这些动作可以先由 agent 提建议，再由人确认执行。
