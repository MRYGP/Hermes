# SK 仓库实战映射

这份文档用于回答：

> Agent OS 的概念，在 SK 仓库里到底拿来干什么？

surface 是当前实现载体，流程链路才是长期知识。

## 四条主链路

### 1. 状态维护链路

目标：

- 发现文档状态漂移。
- 保持看板、任务、产物之间一致。
- 避免旧结论被当成新事实。

当前工具分工：

- Codex：读取本地仓库并输出审计结果。
- Project：保存稳定审计规则。
- Scheduled Task：提醒或监测，但不自动改状态。
- Hermes：提供 cron / memory / compression 的机制对照。

第一版动作：

> 写一个“状态漂移审计”模板，只输出问题清单，不自动改文件。

### 2. 产物发布链路

目标：

- 把阶段成果整理成发布包。
- 检查文档是否完整。
- 生成摘要、变更说明、验收清单。

当前工具分工：

- Work：形成报告、表格、演示稿等候选产物。
- Codex：把确认过的文件写回本地仓库。
- Project：保存稳定发布规则。
- Claude Code：在复杂工程产物中提供第二视角。

第一版动作：

> 先用模板生成发布包清单，再由 Codex 执行文件整理。

### 3. 学习复盘链路

目标：

- 记录学到了什么。
- 判断哪些机制值得迁移到 Codex / Claude Code。
- 形成下一步学习计划。

当前工具分工：

- Project：组织学习上下文和复盘。
- Chat：快速讨论和判断。
- Codex：更新本地文档和状态文件。
- Hermes：提供 Agent OS 机制对照。

第一版动作：

> 每天只记录 3 件事：今天学的概念、在 SK 里的用途、下一步动作。

### 外部信息与动作链路

目标：

- 从 Apps 或其他外部来源读取信息。
- 保留来源、时间和权限边界。
- 把经过验证的结果写回本地仓库。

当前工具分工：

- App：连接外部数据与动作。
- Chat / Work：分析和形成候选产物。
- Codex：把确认结果写回本地仓库。
- Project：保存稳定学习材料。
- Scheduled Task：执行低风险周期提醒或监测。

禁止：

- 未经确认直接把外部结果写成仓库真相。
- 让 Scheduled Task 自动改关键状态。
- 让 App 写操作绕过权限和审批。

## 什么适合做成 skill

适合做成 skill 的动作应该满足：

- 会重复执行。
- 有明确输入输出。
- 有固定步骤。
- 需要少量判断但不需要完全开放推理。

候选：

- 状态漂移审计。
- 收尾联动检查。
- 发布包生成。
- 项目复盘。
- 文档过期检查。

## 什么适合做成 quick command

适合做成 quick command 的动作应该满足：

- 高频。
- 短小。
- 机械。
- 输出格式固定。

候选：

- 打开项目总览。
- 生成今日复盘模板。
- 列出待审计文件。
- 输出固定 commit message 草稿。

## 什么适合做成 cron / automation

适合周期运行的动作应该先满足：

- 只读优先。
- 低风险。
- 可审计。
- 输出清晰。

候选：

- 每日状态漂移检查。
- 每周知识库过期检查。
- 每周工具分工复盘。
- 发布前文档完整性检查。

## 第一阶段不要自动化的事

- 自动重写长文。
- 自动改项目状态。
- 自动发布。
- 自动删除或移动文件。
- 自动修改关键配置。

这些动作可以先由 agent 提建议，再由人确认执行。
