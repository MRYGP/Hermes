# 工具分工对照表

## 总体判断

当前不应把 Hermes 当成主力生产工具。更合理的分工是：

```text
Codex / Claude Code 负责本地执行与复杂工程协作。
ChatGPT Project 负责学习、复盘和稳定知识组织。
Hermes 负责理解 Agent OS 机制如何组织。
```

当前 v2 阶段会优先观察“ChatGPT 工作台”的实际工作方式。“ChatGPT 工作台”是本仓库术语，不是 OpenAI 官方产品名称。

## 当前 surface 选择

| 任务 | 首选 |
| --- | --- |
| 快速讨论、判断、临时解释 | Chat |
| 长任务、研究分析、成品交付 | Work |
| 长期项目上下文和稳定知识 | Project |
| 本地仓库修改、命令和验证 | Codex |
| 复杂工程理解、大范围重构 | Claude Code / Codex |
| Agent OS 机制学习 | Hermes |
| 工作流包装和发现 | Plugin |
| 外部数据、工具和动作连接 | App |
| 周期提醒、简报和监测 | Scheduled Task |

以上是当前阶段的工具选择口径，不是对未来产品形态的永久预测。

## 工具选择口诀

```text
快速讨论，用 Chat。
长任务和成品，用 Work。
长期上下文，用 Project。
修改本地仓库，用 Codex。
复杂工程协作，用 Claude Code。
理解 Agent OS 机制，用 Hermes。
包装和发现工作流，用 Plugin。
连接外部数据与动作，用 App。
周期提醒和监测，用 Scheduled Task。
```

## 推荐分工

### Chat

用于：

- 快速问答
- 即时讨论
- 轻量推理
- 临时解释

不要用它做：

- 长时间仓库执行
- 大范围本地修改
- 自动维护项目状态

### Work

用于：

- 长任务
- 研究分析
- 文档、报告、表格、演示文稿等成品交付
- 人可中途查看、调整和审批的复杂工作

不要用它做：

- 无审批地修改关键仓库文件
- 把生成产物自动视为仓库当前事实
- 未经验证直接发布

### Project

用于：

- 放学习路线
- 放稳定知识文件
- 做复盘
- 做概念对照
- 生成下一步行动

不要用它做：

- 存放高频变化项目状态
- 替代本地仓库真实执行
- 替代 `PROJECT_STATE.md`

### Codex

用于：

- 仓库治理
- 文件修改
- patch 生成
- 测试验证
- PR / review / CI 修复
- 本地自动化执行

不要用它做：

- 长篇抽象理论沉淀的唯一来源
- 频繁变化状态的长期记忆库

### Claude Code

用于：

- 复杂代码理解
- 大范围重构
- 工程协作
- 大代码库问答

不要用它做：

- 无边界的长期管家
- 没有验证闭环的自动执行

### Hermes

用于：

- 学 profile 隔离
- 学 memory 边界
- 学 skills / cron / gateway 组合
- 学 agent 如何长期演化
- 学成本与安全边界

不要用它做：

- 当前阶段的主力仓库执行器
- 生产环境自动改仓库的唯一入口

### Plugin

用于：

- 包装和发现工作流
- 组合 skills、apps 和 templates
- 形成可复用能力入口

不要把 Plugin 和 App 写成同一个概念。

### App

用于：

- 连接外部数据
- 搜索和引用外部来源
- 在审批后执行外部动作

App 返回的数据不是本地仓库真相，必须验证后才能写回。

### Scheduled Task

用于：

- 一次性提醒
- 周期简报
- 低风险监测
- 变化通知

不要用它自动修改关键仓库状态。
