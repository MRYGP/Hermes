# 7 到 14 天学习计划

这个计划不是为了学完 Hermes，而是为了掌握可迁移的 Agent OS 能力。

## 7 天最小版

### 第 1 天：项目定位与工具分工

目标：

- 明确 Hermes 不做主力执行器。
- 明确 Codex / Claude Code / ChatGPT Project / Hermes 的分工。

产出：

- 读完 `docs/00_项目定位.md`
- 读完 `docs/02_工具分工对照表.md`
- 写一段自己的判断：当前为什么还要学 Hermes。
- 记录到 `logs/YYYY-MM/day01_为什么还要学Hermes.md`。

### 第 2 天：AGENTS.md 与项目规则

目标：

- 理解项目规则应该写在哪里。
- 区分项目规则和长期记忆。

产出：

- 起草一个 SK 仓库专用 AGENTS.md 大纲。
- 标记哪些内容不能写入 AGENTS.md。
- 记录到 `logs/YYYY-MM/day02_AGENTS大纲.md`。

### 第 3 天：memory 边界

目标：

- 理解什么能记，什么不能记。

产出：

- 写一份“SK 可长期记忆事实清单”。
- 写一份“禁止写入 memory 的短期状态清单”。
- 记录到 `logs/YYYY-MM/day03_memory边界.md`。

### 第 4 天：skills

目标：

- 理解 skill 是可复用流程，不是长文档。

产出：

- 用 `templates/skill_template.md` 写第一个 skill：状态漂移审计。
- 对照 `examples/01_sk_state_drift_audit_skill.md` 检查是否足够具体。

### 第 5 天：quick commands

目标：

- 理解 quick command 适合机械动作。

产出：

- 用 `templates/quick_command_template.md` 写 2 个 quick command 草案。
- 对照 `examples/02_sk_status_brief_quick_command.md` 检查是否仍是机械动作。

### 第 6 天：automation / cron

目标：

- 理解周期任务应该只读优先。

产出：

- 用 `templates/automation_audit_template.md` 写一个每日状态审计方案。
- 对照 `examples/03_daily_status_drift_audit_automation.md` 确认第一版只读。

### 第 7 天：复盘与取舍

目标：

- 判断哪些能力值得继续学。
- 判断哪些能力已被 Codex / Claude Code 覆盖。

产出：

- 一份 7 天复盘。
- 下一阶段只保留 3 个最值得继续做的动作。
- 记录到 `logs/YYYY-MM/day07_复盘.md`。

## 14 天扩展版

### 第 8 天：hooks / approvals

产出：

- 写出 SK 仓库中需要审批的 5 类动作。

### 第 9 天：compression

产出：

- 写出“会话压缩模板”：当前事实、已完成、未决问题、下一步。

### 第 10 天：fallback model / provider 策略

产出：

- 写出哪些任务必须用强模型，哪些任务可以用低成本模型。

### 第 11 天：profile 隔离

产出：

- 设计 SK profile 与其他项目 profile 的隔离边界。

### 第 12 天：gateway / 常驻 agent

产出：

- 写出 gateway 适用条件与暂缓条件。

### 第 13 天：把流程迁移回 Codex / Claude Code

产出：

- 选择 1 个 Hermes 学到的机制，转写成 Codex / Claude Code 可执行规则。

### 第 14 天：形成 GPTS 草案

产出：

- 修订 `project/SK_Agent_OS_学习台_GPTS_设计稿.md`
- 决定是否需要封装成 GPTS。
