# 7 到 14 天学习计划

> 本计划于 2026-07-10 升级为 v2。
>
> Day 1 已按原计划完成。
> 从 Day 2 开始，当前阶段优先学习 ChatGPT 的实际工作方式，Hermes 机制学习后移，但不取消。

这个计划不是为了学完某个产品，而是为了掌握可迁移的个人 Agent 工作流能力。

## 7 天最小版

### Day 1：项目定位与工具分工

已完成

- 明确 Codex / Claude Code / ChatGPT Project / Hermes 的分工。

### Day 2：ChatGPT 桌面版与 surface 盘点

目标：

- 确认当前安装版本。
- 区分 Chat、Work、Codex。
- 只记录当前账户实际可见能力。
- 不把 rollout 当成已经开放。

产物：

```text
logs/2026-07/day02_ChatGPT新桌面版能力盘点.md
```

### Day 3：Project 与状态同步

目标：

- 理解 Project instructions。
- 理解 Project knowledge。
- 理解 `PROJECT_STATE.md` 当前状态入口。
- 验证 Project 快照和本地仓库的边界。

### Day 4：Chat 与 Work

目标：

- 用同一个真实任务测试 Chat 和 Work。
- 判断何时值得使用 Work。
- 明确 Work 产物何时才会成为仓库事实。

### Day 5：Plugins 与 Apps

目标：

- 区分 Plugin 和 App。
- 实测一个只读能力。
- 检查权限。

### Day 6：Scheduled Tasks

目标：

- 设计一个低风险任务。
- 不自动修改仓库。
- 对比 Scheduled Tasks 和 Codex automations。

### Day 7：第一周复盘

产出：

- 当前最有价值的三个能力。
- 暂时不值得投入的三个能力。
- Hermes 仍值得保留的三个机制。

## 14 天扩展版

### Day 8：AGENTS.md

目标：

- 检查哪些规则应进入仓库级规则。
- 避免把背景百科和当前状态塞进 AGENTS.md。

### Day 9：memory 与 Project context

目标：

- 区分长期事实、Project 上下文和当前状态。
- 写出禁止写入 memory 的短期状态清单。

### Day 10：skills 与 quick commands

目标：

- 选择一个重复流程写成 skill。
- 选择一个机械入口写成 quick command。

### Day 11：permissions、approvals 与安全

目标：

- 写出 SK 仓库中需要审批的 5 类动作。
- 区分 App permissions、Codex approvals 和仓库规则。

### Day 12：compression 与 token 成本

目标：

- 写出会话压缩模板。
- 明确哪些状态应进入 `PROJECT_STATE.md`。

### Day 13：Hermes 机制对照

目标：

- 选择 3 个 Hermes 机制，对照当前 ChatGPT / Codex 工作方式。
- 判断哪些值得迁移成仓库规则或 skill。

### Day 14：个人 Agent 工作流最小版本

目标不是形成大型 Agent OS，而是得到：

```text
一个状态入口
一组稳定规则
一个可复用 skill
一个机械 quick command
一个低风险 task
一套 approval 原则
一份明确工具分工
```
