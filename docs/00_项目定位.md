# 项目定位

## 一句话定位

**SK Agent OS 学习台** 是一个个人 Agent 工作流学习仓库。

它研究的不是某一个具体产品，而是一个人如何长期组织 AI 的上下文、规则、技能、工具、自动化、权限、安全和成本。

## 为什么不是继续叫 SK Hermes 教练

原来的“SK Hermes 教练”定位是：

> 边学 Hermes，边把 SK 仓库用 Hermes 维护和更新。

这个定位在早期合理，但现在需要调整。Codex 和 Claude Code 已经覆盖了很多仓库执行能力，包括读仓库、改文件、跑命令、审查、skills、hooks、MCP、自动化等。

因此 Hermes 不再适合作为默认主力执行层。它更适合作为一个可观察、可拆解、可迁移的 Agent OS 机制样本。

## 当前 v2 阶段

当前阶段优先学习“ChatGPT 工作台”。

“ChatGPT 工作台”是本仓库术语，不是 OpenAI 官方产品名称。它用于组织 Chat、Work、Project、Codex、Plugins、Apps、Scheduled Tasks 等当前能力。

Hermes 保留为 Agent OS 机制教材与对照实验台。

## 稳定主线

```text
SK Agent OS 学习台
= 个人 Agent 工作流学习仓库
+ 以 SK 仓库为真实落地场景
+ 以本地仓库为 source of truth
+ 以 ChatGPT Project 为学习、复盘和知识组织中心
+ 以 Codex / Claude Code 为本地执行与复杂工程协作层
+ 以 Hermes 为 Agent OS 机制教材与对照实验台
```

产品 surface 会变，Agent 工作流要回答的结构问题相对稳定。

## 当前不做什么

- 不把仓库长期定位绑定到某个具体产品形态。
- 不把 OpenAI 当前产品结构当成长期架构。
- 不追求“系统学完 Hermes 全功能”。
- 不把 Hermes 设计成主力仓库执行器。
- 不把 GPTS 做成全自动管家。
- 不把频繁变化的状态放进长期知识库。
- 不做与 SK 无关的大型系统设计。

## 当前只做什么

第一阶段只做 6 件事：

1. 建立个人 Agent 工作流的基本概念框架。
2. 区分长期稳定知识、当前阶段状态和易变能力事实。
3. 学会选择正确 surface：Chat、Work、Project、Codex、Plugin、App、Scheduled Task。
4. 把真实工作映射到 SK 仓库动作。
5. 用 Hermes 理解这些能力背后的 Agent OS 机制。
6. 产出最小可运行模板。

## 判断标准

这个仓库不是为了证明某个产品先进，而是为了回答：

> 如果我要长期使用 AI 维护一个个人知识与项目系统，我应该如何设计记忆、技能、规则、自动化、权限和成本边界？
