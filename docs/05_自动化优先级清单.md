# 自动化优先级清单

## 设计原则

自动化不是越多越好。第一阶段只做能降低重复描述、减少状态漂移、提高执行稳定性的动作。

优先级固定为：

1. 先能跑。
2. 再稳定。
3. 再省 token。
4. 再并行化。
5. 最后才常驻化。

## 三类机制区分

### ChatGPT Scheduled Tasks

适合：

- 提醒。
- 简报。
- 周期检查。
- 变化监测。

Task 的运行结果不会自动成为仓库当前状态。验证后仍需由人或 Codex 更新 `PROJECT_STATE.md`。

### Codex automations

适合：

- 面向 Codex 的固定执行流程。
- 仓库相关任务。
- 带验证闭环的代码或文档检查。

第一版仍应只读优先，不自动改关键状态。

### Hermes cron

适合：

- 理解和实验 agent 调度机制。
- 对照 profile、memory、gateway 与审批边界如何配合。

当前不把 Hermes cron 作为主力生产自动化入口。

## P0：状态漂移审计

目标：

- 找出文档之间的状态不一致。
- 找出过期结论。
- 找出缺少下一步动作的任务。

第一版范围：

- 只读。
- 只输出问题。
- 不自动改文件。

推荐载体：

- skill
- automation audit template

## P1：启动会话摘要

目标：

- 每次开始工作前快速知道当前上下文。
- 减少重复解释。
- 避免读错过期状态。

第一版范围：

- 项目是什么。
- 当前阶段是什么。
- 本次最值得做什么。
- 哪些文件不能被当成最新状态。

推荐载体：

- quick command
- ChatGPT Project instructions

## P1：收尾联动检查

目标：

- 每次完成任务后检查是否需要更新 README、看板、计划、变更记录。

第一版范围：

- 输出检查清单。
- 不自动写入。

推荐载体：

- skill
- hook 的前置设计稿

## P1：发布包生成

目标：

- 把阶段产物整理成可交付结构。

第一版范围：

- 生成发布包目录建议。
- 生成摘要和验收清单。
- 文件操作交给 Codex 或人工确认。

推荐载体：

- skill

## P2：知识库过期检查

目标：

- 防止 ChatGPT Project / GPTS 知识库中保留过期状态。

第一版范围：

- 检查哪些文件适合长期放入知识库。
- 标记高频变化文件。

推荐载体：

- skill
- 周期性只读 automation

## P2：雷达扫描

目标：

- 周期性扫描外部信息或竞品信息。

第一版范围：

- 只做人工触发。
- 不做自动写入。

推荐载体：

- ChatGPT Project 对话
- Codex / browser 辅助

## P3：gateway / cron / 多 worktree

目标：

- 让 agent 常驻或并行处理任务。

当前判断：

> 暂缓。等 P0-P2 跑顺后再做。

原因：

- 配置复杂度上升。
- 排错成本上升。
- 容易过早自动化错误流程。

## 当前最小闭环

```text
手动学习一个概念
-> 映射到 SK 动作
-> 写成模板
-> 用 Codex / Claude Code 执行一次
-> 复盘是否值得沉淀为 skill / quick command / automation
```
