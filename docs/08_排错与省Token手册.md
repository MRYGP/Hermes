# 排错与省 Token 手册

## 常见排错顺序

遇到 agent 行为不稳定时，不要先怀疑模型。按这个顺序查：

1. 当前任务是否说清楚。
2. 项目规则是否过长或过旧。
3. 关键文件是否被读到。
4. 记忆中是否有过期事实。
5. skill 是否过于抽象。
6. quick command 是否承担了复杂判断。
7. 自动化是否过早写入。
8. 是否缺少审批或 hook。

## AGENTS.md 太长怎么办

处理原则：

- 只保留 agent 每次进入仓库都必须知道的规则。
- 背景材料移到 docs。
- 高频状态移到看板。
- 长流程移到 skills。
- 机械动作移到 quick commands。

理想 AGENTS.md 应该回答：

- 这个项目是什么。
- 文件怎么组织。
- 修改时不能碰什么。
- 改完要怎么验证。
- 输出风格是什么。

## skill 不生效怎么办

检查：

- skill 触发场景是否明确。
- 输入输出是否清楚。
- 步骤是否可执行。
- 是否混入太多背景知识。
- 是否把一次性任务写成了技能。

修正：

- 删除长背景。
- 增加输入要求。
- 增加验收标准。
- 增加“不要做什么”。

## memory 混乱怎么办

检查：

- 是否把短期状态写进 memory。
- 是否把旧判断当成长期事实。
- 是否多个项目共用同一套记忆。

修正：

- 只保留长期偏好和稳定原则。
- 把状态迁回仓库文档。
- 用 profile / project 做隔离。

## automation 误触发怎么办

检查：

- 是否一开始就允许写入。
- 是否缺少审批。
- 是否缺少日志。
- 是否没有限定文件范围。

修正：

- 第一版改成只读。
- 输出报告，不直接修改。
- 高风险动作必须人工确认。

## 省 Token 基本原则

1. 不把长背景塞进每次提示词。
2. 不把短期状态写进长期知识库。
3. 让 AGENTS.md 只写规则。
4. 让 skills 只写流程。
5. 让 quick commands 承担机械动作。
6. 长会话后做结构化压缩。
7. 每次只让 agent 读与任务相关的文件。

## 会话压缩模板

```text
当前目标：

已完成：

关键决策：

当前事实：

未决问题：

下一步动作：

不要重复做：
```

## 模型使用建议

强模型用于：

- 架构判断
- 大范围重构
- 复杂审计
- 高风险修改

低成本模型用于：

- 固定格式摘要
- 文档清单
- 简单分类
- 机械检查

不要为了省成本，把高风险判断交给弱模型。
